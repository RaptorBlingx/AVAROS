## 🎯 Objective

Build a complete voice-controlled system where OVOS commands control real devices through an API backend. This simulates how AVAROS will interact with manufacturing platforms like RENERYO.

**What you're proving:** OVOS can control external systems via voice, not just answer questions.

---

## 📋 Prerequisites

✅ Weather skill task completed  
✅ Understanding of OVOS intents, dialogs, and API calls

---

## 🏗️ What You're Building

### Two Components:

1. **Mock Device API** (Backend)
   - Simulates an IoT/manufacturing device management platform
   - Manages state of multiple devices (machines, sensors, equipment)
   - Responds to control commands (on/off/status queries)

2. **OVOS Device Control Skill** (Frontend)
   - Voice commands to query device status
   - Voice commands to control devices (turn on/off)
   - Real-time feedback via spoken responses

---

## 🎤 Required Voice Interactions

Your system must support these conversations:

```
User: "Hey Mycroft"
OVOS: [beep]

User: "What's the status of device 1?"
OVOS: "Device 1 is online and currently running"

---

User: "Hey Mycroft, turn off machine 3"
OVOS: "Turning off machine 3... Done. Machine 3 is now off"

---

User: "Hey Mycroft, turn on light 2"  
OVOS: "Turning on light 2... Done. Light 2 is now on"

---

User: "Hey Mycroft, list all devices"
OVOS: "You have 5 devices: device 1 is on, machine 3 is off, light 2 is on, sensor 4 is online, and device 5 is in maintenance"
```

---

## 📊 Mock Devices Requirements

Your API should simulate **at least 5 devices** with generic names:
- Device 1, Device 2, ...
- Machine 1, Machine 2, ...
- Light 1, Light 2, ...
- Sensor 1, Sensor 2, ...

Each device should have:
- Unique ID (e.g., `device-1`, `machine-3`, `light-2`)
- Status states (online, offline, maintenance)
- Power states (on, off)
- Realistic metrics (power consumption, temperature, operating hours, etc.)

**Example device:**
```
Device 1:
- Status: online
- Power: on  
- Metrics: 45.2 kW consumption, 68°C temperature, 1248 operating hours
```

---

## ✅ Success Criteria

### Functional
- [ ] Voice interaction works end-to-end (wake word → command → action → response)
- [ ] Devices respond to control commands (state changes persist)
- [ ] Status queries return current device state
- [ ] System handles invalid commands gracefully ("Sorry, device 99 not found")

### Technical
- [ ] Zero-config deployment via Docker Compose
- [ ] OVOS skill communicates with API over REST
- [ ] State management (turning device on/off actually changes its state)
- [ ] Error handling (device not found, already in requested state, API down)

### Documentation
- [ ] README explains setup and usage
- [ ] Test results documented (at least 7 different commands tested)
- [ ] Architecture explanation (how OVOS talks to API)
